# clear-thinking 路由器

本文件负责把用户问题转成合适的微技能分类和组合路径。它不是关键词分类表，而是内部决策器：先判断问题处境，再只读取命中的分类微技能文件，达到可行动条件就停止分析。

## 1. 路由原则

1. **先处境，后方法**：先判断用户卡在目标、事实、结构、取舍、执行、验证还是复盘。
2. **先索引，后内容**：需要微技能时，先看 `MICRO_SKILLS.md` 或本文件的分类表，再只读命中的 `micro-skills/*.md`。
3. **先事实，后框架**：事实不足时先定位最小事实缺口；框架只提供观察角度，不提供现成答案。
4. **先主要矛盾，后建议清单**：建议必须围绕本轮最关键张力，不平均展开。
5. **先最小充分，后深度展开**：能用 1 个微技能解决，不调用 3 个；能读 1 个分类文件，不读全部分类。
6. **先停止条件，后继续分析**：达到可行动、需工具、需用户确认、需子代理或需 Workflow 的条件，就停止空想。

## 2. 内部路由流程

```text
感知输入 → 诊断处境 → 选择分类文件 → 选择微技能 → 组合路径 → 输出并检查
```

### 2.1 感知输入

```markdown
- 用户表层请求是什么？
- 用户真正想改变的状态是什么？
- 已知事实有哪些？
- 哪些是推断或待确认假设？
- 当前硬约束是什么？
- 本轮是否要落盘、修改文件、调研、复盘或完成检查？
```

### 2.2 诊断处境并选择文件

| 处境 | 判断信号 | 读取文件 | 候选微技能 |
| --- | --- | --- | --- |
| 目标/事实/结构/取舍/执行/验证 | 目标不清、事实不足、问题很多、需要行动或完成检查 | `micro-skills/common.md` | 目标澄清、事实/推断/假设分离、问题分型、主要矛盾、最小行动、风险扫描、信息缺口、方案取舍、完成检查 |
| 产品设计 | 需要定义用户、场景、流程、字段、状态、权限、验收 | `micro-skills/product-rd.md` | 需求四问、复用/新建判断、信息一二三档、页面范式、字段/状态/操作卡、质量裁决 |
| AI / 组织产品 | 涉及 Agent、工作台、默认值、已读、卡片、组织协同 | `micro-skills/ai-org-product.md` | 发心拆层、定位四问、不可能三角、组件权力、责任状态、默认值制度、共创偏差、敏捷债务 |
| 创业经营 / 长期判断 | 涉及方向、增长、效率、口碑、长期主义、生态协同 | `micro-skills/business-long-term.md` | 顺势判断、用户朋友、爆品聚焦、效率革命、口碑飞轮、长期复利、生态协同 |
| 高上下文内耗 | 分析很多但难行动，或把痛感不断结构化 | `micro-skills/high-context.md` | 痛感分流、解释关闭、切断力 |
| 技能建设 / 方法吸收 | 要从材料中抽象方法、封装微技能、优化本技能 | `micro-skills/method-absorption.md` | 判断动作提取、微技能封装、反权威背书 |

读取规则：

- 绝大多数任务只读 `common.md` 或 `common.md` + 一个专项文件。
- 不要因为用户出现某个领域词，就直接读对应专项文件；必须先判断是否会改变本轮行动。
- 如果需要 3 个以上分类文件，优先考虑子代理或 Workflow，而不是在主会话吞下全部上下文。

## 3. 微技能选择

选择时内部回答：

```markdown
- 本轮最少需要哪些微技能？
- 它们分别来自哪个分类文件？
- 为什么选择它们？
- 哪些看似相关的方法本轮不用？
- 如果判断错了，最先出现什么信号？
```

数量控制：

| 任务复杂度 | 微技能数量 | 处理方式 |
| --- | --- | --- |
| 低 | 0-2 个 | 内部想清楚后直接执行或短答 |
| 中 | 3-5 个 | 标准组合，输出主要判断和行动 |
| 高 | 5 个以上 | 拆子代理或 Workflow，不在主会话长篇展开 |

## 4. 组合路径

### 4.1 轻量组合

```text
目标澄清 → 最小行动
```

适用：低风险、目标基本明确、材料少。

### 4.2 标准组合

```text
事实/假设分离 → 主要矛盾 → 1-3 个专项微技能 → 最小行动 → 验证
```

要求：只选 1 个主要矛盾，明确暂不处理项。

### 4.3 深度组合

```text
现实校准 → 方法吸收/问题重构 → 反教条检查 → 微技能重组 → 落地执行 → 验证沉淀
```

适用：多材料、多阶段、用户要求“不要浅”、需要沉淀技能或方法论。多材料优先子代理读取，主会话只接收摘要。

### 4.4 专项组合

| 专项 | 常用组合 | 裁剪方式 |
| --- | --- | --- |
| 产品研发 | 需求四问 → 复用/新建判断 → 信息一二三档 → 页面范式 → 字段/状态/操作卡 → 质量裁决 | 只问需求真假时停在需求四问；只审查交付时直接质量裁决 |
| AI / 组织产品 | 发心拆层 → 定位四问 → 不可能三角 → 组件权力 → 责任状态 → 默认值制度 → 用户证据校准 → 敏捷债务 | 只讨论方向时用发心+定位；只讨论交互时用组件+状态+默认值 |
| 创业经营 | 顺势判断 → 用户朋友 → 爆品聚焦 → 效率革命 → 口碑飞轮 → 长期复利 | 判断方向用顺势+用户；判断增长用口碑+实验；判断长期用长期复利 |
| 高上下文降噪 | 痛感分流 → 解释关闭 → 切断力 → 最小行动 | 不做心理诊断，不把痛苦浪漫化，明确哪些责任不承担 |
| 复盘 | 目标 → 结果 → 偏差 → 主要原因 → 下一轮最小改法 → 沉淀判断 | 不写流水账，必须给偏差和下一轮动作 |

### 4.5 并行组合

当任务可拆为互不依赖的专项时，优先子代理并行：

```text
材料深读 + 外部资料研究 + 现有架构审查 + 风险审查 → 主会话整合
```

### 4.6 迭代组合

```text
假设 → 最小行动/实验 → 反馈 → 修正 → 下一轮
```

每轮必须回答：当前假设、最小验证、成功信号、失败信号、下一轮调整。

## 5. 缺口路由

| 缺口类型 | 处理方式 |
| --- | --- |
| 目标缺口 | 用“目标澄清”重写目标；如果仍依赖用户偏好，再追问 |
| 事实缺口 | 先查工作区索引、命中记录、相关文件或公开资料 |
| 历史缺口 | 先读 `.claude/skills/work-journal/resources/index.md`，只读命中记录 |
| 材料缺口 | 子代理读取最小材料并返回摘要，主会话不吞长上下文 |
| 结构缺口 | 用“问题分型/系统关系/主要矛盾”而不是直接给建议 |
| 取舍缺口 | 用“方案取舍”，明确不选什么 |
| 执行缺口 | 生成 3-7 步以内的最小行动 |
| 验证缺口 | 用“最小验证/失败信号/完成检查” |

## 6. 升级与停止条件

### 6.1 可以停止分析并行动

- 目标已经明确；
- 下一步行动明确；
- 风险可接受；
- 验收方式明确；
- 信息缺口不会改变本轮行动。

### 6.2 需要工具

- 文件事实可查；
- 需要外部最新资料；
- 需要运行结果、截图或页面状态；
- 需要验证路径是否存在。

### 6.3 需要子代理

- 需要读取 2 份以上长材料；
- 需要并行研究多个信息源；
- 需要独立视角审查；
- 分析会显著膨胀主会话上下文。

### 6.4 需要 Workflow

- 跨阶段、跨项目、多产物；
- 需要“定位记录 → 分析 → 执行 → 验证 → 回写”的闭环；
- 单个会话难以稳定协调；
- 需要多个子代理交叉检查。

### 6.5 需要追问用户

只有满足以下条件才追问：

- 缺口只有用户能回答；
- 继续假设会明显偏离用户真实意图；
- 方案取舍依赖用户价值偏好；
- 涉及不可逆或对外发布动作。

## 7. 反关键词匹配

禁止这样做：

```text
用户说产品 → 套产品清单
用户说创业 → 套经营清单
用户说痛苦 → 套高上下文卡
用户说某材料 → 套材料观点
```

应该这样做：

```text
先判断当前问题到底缺什么：事实、目标、结构、取舍、行动还是验证。
再只读取命中的分类文件。
最后只输出与当前问题有关的判断和最小行动。
```
