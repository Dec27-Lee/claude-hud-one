# clear-thinking

当前工作区的思考辅助技能。它不是资料汇编、来源索引或提示词清单，而是一个轻量的“微技能操作系统”：

```text
事实校准 → 处境诊断 → 微技能选择 → 组合推理 → 最小行动 → 验证修正
```

## 0. 核心定位

- **负责判断**：事实是什么、真实目标是什么、主要矛盾是什么、下一步如何验证。
- **负责组合**：按问题处境选择最小充分微技能，不按“产品/增长/创业/人生”等关键词套模板。
- **负责降噪**：能行动就停止分析；缺事实就查证；缺偏好才追问；多材料/多阶段才升级子代理或 Workflow。
- **不负责记录**：跨轮需求、继续历史事项、进度、复盘和完成检查仍用 `.claude/skills/work-journal/SKILL.md`。
- **不替代调度**：是否使用 Workflow / 子代理，最终以根目录 `CLAUDE.md` 和 `work-journal` 规则为准。
- **不替代专业技能**：代码、Figma、安全、Claude API 等有专用技能时，本技能只做前置判断、方案取舍或复盘辅助。
- **不保留来源噪声**：素材、书籍和案例只在建设阶段被吸收为判断动作；运行时不按来源读取、引用或背书。

## 1. 资源文件

默认只读 `SKILL.md`。确实需要展开时，再按最小必要读取：

| 文件 | 何时读取 | 用途 |
| --- | --- | --- |
| `.claude/skills/clear-thinking/resources/ROUTER.md` | 需要判断问题处境、选择组合路径时 | 路由、规划、组合、停止条件 |
| `.claude/skills/clear-thinking/resources/MICRO_SKILLS.md` | 需要定位具体微技能分类时 | 微技能索引，不存放完整内容 |
| `.claude/skills/clear-thinking/resources/micro-skills/*.md` | 已确定问题处境后 | 分类微技能内容，只读命中的 1-2 个文件 |
| `.claude/skills/clear-thinking/resources/CHECKLISTS.md` | 完成前检查、用户要求“别浅/别教条/检查是否靠谱”时 | 输出质量、反教条、产品/经营/长期判断检查 |

禁止为了“显得有依据”读取或复述素材来源。用户问依据时，只解释当前判断依赖的事实、假设和验证方式；不要用书名、人物、公司案例当权威。

## 2. 触发场景

优先用于：

- 用户要求分析、判断、规划、复盘、取舍、找关键问题。
- 用户指出回答“浅”“像提示词”“没提炼方法论”“太教条”。
- 当前问题涉及多目标冲突、长期后果、系统约束，或用户/组织/技术/商业之间的张力。
- 需要先决定“做什么、不做什么、先做哪一步、怎么验收”。

不要用于：

- 单点明确小任务、一次性短答、低风险直接操作。
- 用户明确要求不要分析、只执行。
- 已有专用技能必须优先处理的任务；此时 clear-thinking 只做前置判断。

## 3. 执行协议

内部执行，不必完整展示：

```text
1. 感知输入：事实、目标、约束、缺口、风险。
2. 处境路由：判断当前卡在目标、事实、结构、取舍、执行、验证还是复盘。
3. 微技能选择：先用 `MICRO_SKILLS.md` 定位分类，再只读取命中的 1-2 个分类文件。
4. 组合推理：从已读取分类中选择 1-5 个最小充分微技能，按轻量/标准/深度/专项/并行/迭代路径组合。
5. 反教条检查：确认不是关键词匹配、框架表演或权威背书。
6. 输出裁剪：只输出当前判断、最小行动、验证和风险。
```

复杂问题内部先形成这张卡：

```markdown
- 问题处境：
- 主要矛盾：
- 关键缺口：
- 本轮采用微技能：
- 本轮不用的微技能：
- 组合顺序：
- 停止条件：
- 验证方式：
```

只有当用户要求解释方法时，才展示规划依据。

## 4. 反关键词匹配

禁止：

```text
用户说产品 → 套产品清单
用户说创业 → 套创业方法
用户说痛苦 → 套高上下文卡
用户说某本书/某材料 → 套来源观点
```

必须：

```text
先判断当前到底缺事实、目标、结构、取舍、行动还是验证；再选择微技能组合。
```

## 5. 输出门槛

触发本技能后的合格输出至少满足：

1. **有当前判断**：不能只列框架。
2. **有事实边界**：区分事实、推断、假设和价值偏好。
3. **有主要矛盾**：说明本轮最关键的张力，不平均展开。
4. **有选择依据**：复杂任务内部说明为什么选这些微技能、不选那些。
5. **有最小行动**：给下一步、验收方式、失败信号和后手。
6. **有反教条检查**：不让关键词、模板、书名、案例替代当前问题。
7. **有调度边界**：判断是否需要 work-journal、工具、子代理、Workflow、workspace-index 更新。

默认短输出：

```markdown
## 结论
## 依据与假设
## 主要矛盾
## 最小行动与验证
## 风险与边界
```

如用于工作日志回写，压缩为：

```markdown
## clear-thinking 摘要
- 真实目标：
- 主要矛盾：
- 微技能组合：
- 本轮最小行动：
- 验收方式：
- 风险/失败信号：
- 是否需要工具/子代理/Workflow：
```

## 6. 与工作区其他机制的配合

- 涉及“记录、继续、上次、之前、进度、复盘、完成检查”时，先按 `work-journal` 规则通过索引定位并判断是否需要记录，再用本技能辅助判断。
- 涉及大任务、多材料、多阶段、需要交叉验证时，本技能只定义拆分边界和验收标准；是否使用 Workflow / 子代理以及如何编排，按 `CLAUDE.md` 和 `work-journal` 规则执行。
- 涉及找资料时，先读 `.claude/workspace-index.md`；涉及历史记录时，先读 `.claude/skills/work-journal/resources/index.md`，命中后才读对应 record。
- 新增、删除、移动、重命名长期资料入口时，完成前检查是否需要更新 `.claude/workspace-index.md`。

## 7. 禁忌

- 不要用鸡汤、宏大口号、书名背书替代具体路径。
- 不要伪造书籍章节、页码、原文引文。
- 不要把小说人物判断、政治文本语境、商业案例机械搬到软件工程、产品研发或经营决策。
- 不要把增长理解成作弊式拉新；不能牺牲隐私、信任、长期留存和用户尊严。
- 不要为了避免浅薄而把所有任务都变成重流程。
- 不要把材料里的思想压缩成“关键词 → 方法名 → 模板”的调用链。
- 不要为了显示自己读过材料而复述材料；材料只用于训练和维护微技能，运行时不输出来源。
